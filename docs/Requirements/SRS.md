# 1. Введение

<a name="appointment"/>

## 1.1. Назначение
В этом документе описаны функциональные и нефункциональные требования к веб-приложению для управления кулинарными рецептами и инвентарем "KitchenMate". Документ предназначен для команды разработки, преподавателя и заинтересованных сторон для обеспечения единого понимания целей и возможностей продукта.

<a name="business_requirements"/>

## 1.2. Бизнес-требования

<a name="initial_data"/>

### 1.2.1. Исходные данные
В современном ритме жизни планирование питания часто занимает много времени, а неиспользованные продукты приводят к пищевым отходам. Пользователи сталкиваются с проблемами: покупают лишние ингредиенты, забывают, что есть дома, или не могут быстро найти подходящий рецепт. Необходим простой инструмент для управления продуктами и поиска рецептов на основе имеющихся ингредиентов.

<a name="business_opportunities"/>

### 1.2.2. Возможности
Приложение "KitchenMate" предоставляет удобный инструмент для управления продуктами, поиска рецептов и создания списков покупок. Это помогает экономить время на планирование еды, минимизировать отходы и упростить готовку. Простота интерфейса и интеграция с внешним API делают приложение доступным для пользователей с базовыми техническими навыками.

<a name="project_boundary"/>

### 1.2.3. Границы проекта
KitchenMate — веб-приложение, которое позволяет пользователям добавлять продукты в виртуальную кладовую, искать рецепты по имеющимся ингредиентам через Spoonacular API и генерировать списки покупок. В текущей версии не предусмотрены функции сканирования штрих-кодов, анализа калорий, регистрации пользователей или синхронизации данных. Все данные хранятся локально в SQLite.

<a name="analogues"/>

## 1.3. Аналоги
На рынке существуют приложения для планирования еды, такие как Yummly, BigOven и Mealime. KitchenMate отличается простотой, отсутствием необходимости регистрации и фокусом на использовании имеющихся продуктов без дополнительных функций, таких как социальные взаимодействия или сложные диеты.

<a name="user_requirements"/>

# 2. Требования пользователя

<a name="software_interfaces"/>

## 2.1. Программные интерфейсы
Приложение взаимодействует со следующими системами:  
Spoonacular API для поиска рецептов по ингредиентам (REST, JSON, https://spoonacular.com/food-api).  
SQLite для локального хранения данных кладовой (название продукта, количество).  
Взаимодействие с браузерами (Chrome, Safari, Firefox) для отображения веб-интерфейса.

<a name="user_interface"/>

## 2.2. Интерфейс пользователя
Интерфейс приложения состоит из четырех основных экранов:  

1. Главный экран  
   Отображает поле для ввода ингредиентов, кнопку "Найти рецепты" и ссылку на кладовую.  
   Позволяет начать поиск рецептов или перейти к управлению продуктами.  

2. Экран кладовой  
   Содержит таблицу с продуктами (название, количество), кнопки для добавления и удаления продуктов.  
   Реализована форма для ввода названия и количества продукта.  

3. Экран рецептов  
   Отображает список рецептов с указанием названия, имеющихся и недостающих ингредиентов.  
   Содержит кнопку для генерации списка покупок.  

4. Экран списка покупок  
   Показывает чек-лист недостающих ингредиентов с возможностью отметить купленные продукты.  

Таблица "Действие пользователя - Реакция":  
| Действие пользователя | Реакция системы |  
| Вводит ингредиенты и нажимает "Найти рецепты" | Отображает список рецептов из Spoonacular API с указанием наличия ингредиентов |  
| Нажимает "Добавить продукт" в кладовой | Открывает форму для ввода названия и количества |  
| Выбирает рецепт и нажимает "Создать список покупок" | Генерирует чек-лист недостающих ингредиентов |  
| Отмечает продукт в списке покупок | Обновляет статус продукта (куплено/не куплено) |

<a name="user_specifications"/>

## 2.3. Характеристики пользователей

<a name="application_audience"/>

### 2.3.1. Аудитория приложения
Целевой аудиторией являются люди, планирующие питание:  
Студенты (18–25 лет, средняя техническая грамотность, цель — бюджетная готовка).  
Семьи/домохозяйки (25–50 лет, базовая техническая грамотность, цель — экономия времени).  
Вегетарианцы (18–40 лет, средняя/высокая техническая грамотность, цель — рецепты без мяса).  
Пользователи обладают базовыми навыками работы с веб-интерфейсами.

<a name="assumptions_and_dependencies"/>

## 2.4. Предположения и зависимости
Приложение рассчитано на одного пользователя с доступом к интернету для API-запросов.  
Корректная работа зависит от доступности Spoonacular API (150 запросов/день на бесплатном тарифе).  
Пользователь должен знать, какие продукты есть дома, для ввода в кладовую.

# 3. Системные требования

<a name="functional_requirements"/>

## 3.1. Функциональные требования

<a name="main_functions"/>

### 3.1.1. Основные функции

<a name="pantry_management"/>

#### 3.1.1.1. Управление кладовой
Описание: Пользователь может полностью контролировать список продуктов в кладовой.  

| Функция | Требования |  
| Добавление продукта | Приложение должно предоставлять форму для ввода названия и количества продукта |  
| Удаление продукта | Пользователь должен иметь возможность удалить продукт из кладовой |  
| Просмотр кладовой | Система отображает таблицу с названием и количеством продуктов |  

<a name="recipe_search"/>

#### 3.1.1.2. Поиск рецептов
Описание: Пользователь может искать рецепты на основе ингредиентов.  

| Функция | Требования |  
| Поиск по ингредиентам | Система запрашивает рецепты через Spoonacular API, используя список ингредиентов из кладовой или введенных вручную |  
| Отображение рецептов | Система показывает рецепты с указанием, какие ингредиенты есть, а какие нужно купить |  

<a name="shopping_list"/>

#### 3.1.1.3. Генерация списка покупок
Описание: Пользователь может создать список недостающих ингредиентов.  

| Функция | Требования |  
| Создание списка покупок | Система генерирует чек-лист недостающих ингредиентов для выбранного рецепта |  
| Отметка покупок | Пользователь может отмечать купленные продукты в чек-листе |  

<a name="restrictions_and_exclusions"/>

### 3.1.2. Ограничения и исключения
Приложение не поддерживает сканирование штрих-кодов.  
Отсутствует анализ калорий или диетических предпочтений.  
Нет системы регистрации или облачной синхронизации данных.  

<a name="non-functional_requirements"/>

## 3.2. Нефункциональные требования

<a name="quality_attributes"/>

### 3.2.1. Атрибуты качества

<a name="requirements_for_ease_of_use"/>

#### 3.2.1.1. Требования к удобству использования
Интерфейс должен быть интуитивно понятным, без перегруженности элементами.  
Все кнопки и поля ввода должны иметь четкие названия или иконки.  
Навигация между экранами должна быть простой (например, кнопки "Назад", "Добавить").  

<a name="performance_requirements"/>

#### 3.2.1.2. Требования к производительности
Время ответа API на поиск рецептов должно быть менее 3 секунд.  
Отрисовка страниц и обновление кладовой должны происходить без заметных задержек.  
Приложение должно корректно загружаться в браузере за 2 секунды.  

<a name="data_storage_requirements"/>

#### 3.2.1.3. Требования к хранению данных
Данные кладовой хранятся в локальной базе SQLite в структурированном формате.  
Система должна обрабатывать ошибки чтения/записи базы, информируя пользователя.  
Данные должны сохраняться при закрытии приложения без потерь.  

<a name="external_interfaces"/>

### 3.2.2. Внешние интерфейсы
Интерфейс разработан с использованием HTML5/CSS3 для поддержки современных браузеров (Chrome, Safari, Firefox).  
Размер шрифтов и элементы интерфейса подобраны для комфортного чтения.  

<a name="restrictions"/>

### 3.2.3. Ограничения
Приложение реализовано как веб-приложение с использованием REST API (Spoonacular).  
Тестирование проводится в современных браузерах (Chrome 100+, Safari 15+).  
Лимит API-запросов: 150 в день (бесплатный тариф Spoonacular).
